# Database

Vector uses two Postgres databases:

| Database | Purpose |
|---|---|
| `mmff_vector` | Tenant data — subscriptions, users, nav, portfolio models, items |
| `mmff_library` | MMFF-authored content — portfolio model definitions, terminology |

## Tenant isolation

Every tenant row carries a `subscription_id` FK. All queries must filter by the authenticated user's `subscription_id`. The Go service layer enforces this — raw DB access must follow the same rule.

## Soft-archive pattern

Deletions are soft. Tables with user-visible data carry an `archived_at TIMESTAMPTZ` column. Live queries always filter `archived_at IS NULL`.

## Migrations

Migrations live in `db/schema/` as numbered SQL files (`001_...sql`, `002_...sql`, …). Run with:

```bash
go run ./backend/cmd/migrate [-dry-run] [-db vector|library|both]
```
