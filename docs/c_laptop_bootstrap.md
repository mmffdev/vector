# Laptop bootstrap — "clone, pull, go"

> Last verified: 2026-04-24

Goal: open a fresh laptop (or second Mac), pull from remote, and be ready to work within one command.

## One-command bootstrap

```bash
cd "/path/to/MMFFDev - PM"
./dev/scripts/bootstrap_laptop.sh
```

It runs three things in order, all idempotent:

1. **`link_memory.sh`** — symlinks `~/.claude/projects/…/memory` → repo's `.claude/memory/`. Any auto-memory update on any laptop travels through git to every other laptop.
2. **`ssh_manager.sh`** — installs brew, libpq, autossh, node@20; generates SSH key; pushes it to `mmffdev.com`; appends the three `mmffdev-*` aliases to `~/.ssh/config`; starts the tunnel; writes `backend/.env.local`; runs `npm install`; verifies DB round-trip.
3. **`install_launchagent.sh`** (asks first) — installs `com.mmff.tunnel.plist` so the tunnel auto-starts at login and restarts on failure. Logs at `~/Library/Logs/mmff/tunnel.*.log`.

## What's committed vs. per-laptop

| Thing | Committed | Per-laptop |
|---|---|---|
| Auto-memory (`.claude/memory/`) | ✅ | symlink created by `link_memory.sh` |
| `backend/.env.local` (credentials) | ✅ (private repo) | pulled via git |
| Docs + CLAUDE.md | ✅ | pulled via git |
| SSH key `~/.ssh/id_ed25519` | ❌ | generated by `ssh_manager.sh` |
| `~/.ssh/config` aliases | ❌ | appended by `ssh_manager.sh` |
| brew packages | ❌ | installed by `ssh_manager.sh` |
| LaunchAgent plist | ❌ (template is) | installed by `install_launchagent.sh` |
| Tunnel process | ❌ (ephemeral) | started by launchagent at login |

## Auto-memory across laptops

`.claude/memory/` is a real directory in the repo. The symlink at
`~/.claude/projects/-Users-rick-Documents-MMFFDev-Projects-MMFFDev---PM/memory`
points at it. Claude Code reads/writes through that symlink, so:

- Saving a memory on laptop A → writes a file in the repo → `git add` + commit.
- Pulling on laptop B → file lands in repo → symlink surfaces it instantly.

The symlink path is identical on any Mac whose username is `rick` and which has the repo at `~/Documents/MMFFDev-Projects/MMFFDev - PM`. If either differs, update `link_memory.sh`.

**Backup:** the first run renames the pre-symlink directory to
`memory.pre-symlink-backup.<timestamp>` rather than deleting it. Keep the most recent as a safety net; older ones can be removed.

## LaunchAgent — auto-tunnel at login

The plist (`dev/scripts/launchagent/com.mmff.tunnel.plist.template`) runs `ssh -N mmffdev-pg` under `launchd` with `KeepAlive` scoped to:

- `SuccessfulExit: false` — if ssh exits non-zero, relaunch.
- `NetworkState: true` — wait for the network before relaunch.
- `ThrottleInterval: 30` — don't busy-loop on repeated failures.

Manage:

```bash
launchctl bootout gui/$(id -u)/com.mmff.tunnel              # stop
launchctl bootstrap gui/$(id -u) ~/Library/LaunchAgents/com.mmff.tunnel.plist  # start
tail -f ~/Library/Logs/mmff/tunnel.err.log                  # watch
```

## If something goes wrong

- **`link_memory.sh` moved the dir but didn't symlink** — restore from `memory.pre-symlink-backup.<timestamp>` and re-run the script.
- **Tunnel won't come up** — `~/Library/Logs/mmff/tunnel.err.log` usually shows why (`Permission denied`, `Address already in use`, etc.). Most common: stale tunnel from a previous session still holding port 5434. `pkill -f 'ssh -N.*mmffdev-pg'` then reload the agent.
- **Adminer 404** — `nc -z localhost 8081` first. If that fails the tunnel is stale; if it succeeds but the page won't load, the container on the server may be down (`ssh mmffdev-admin 'docker ps --filter name=adminer'`).
