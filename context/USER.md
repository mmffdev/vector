## Preferences

**Stakeholder foundation mode** (sole stakeholder, no deadline, 2026-05-16):
- **Foundation > patch.** "Is this right?" before "is this fast?". Option B (right architecture, bigger PR) usually beats Option A (smallest correct thing).
- **POC = real architecture, narrowly wired** — not throwaway.
- **Name scope creep as scope correction.** "Fix the bug" hiding three structural issues → say so up front.
- **Tech-debt is a flag, not an exit.** Never substitute for finding the right answer.
- **Cleanup, not deferral.** Two-hour fix over one-hour hack.

## Working Style

**Design conversation IS the iteration loop** (2026-05-16). No customer-feedback cycle to optimise for; design discussion replaces it.
- **Don't rush to code.** "How would this work?" usually wants thinking, not a half-built prototype.
- **Push back honestly.** Bland agreement is less useful than honest disagreement. "Honest read", "let me play that back", "where I'd push back gently".
- **Play back proposals** before responding. Rick types fast/short — meaning often needs inference ("ncy" = "nice catch yes").
- **Surface tensions, don't paper over.** Two preferences in conflict → name it.
- **Long-form earns its length.** Brief when the question is brief; structured on architectural choices.
- **Converge before coding.** "Yes, that's the shape" → that's the moment to code.

**Always recommend the safest, best approach.** Never neutral A/B/C menus. Lead with explicit recommendation: "Recommended: X. Alternative: Y. Avoid: Z." If only one safe option exists, say so plainly.
