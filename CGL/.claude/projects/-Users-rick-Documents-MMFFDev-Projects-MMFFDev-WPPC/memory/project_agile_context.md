The sprint system, backlog management, user stories, changelog, and planning pages serve dual purpose:

1. **Showcase asset** — demonstrates the platform's project management capabilities to potential users/customers
2. **Practical tool** — genuinely helps track and organise work, even if it's not textbook Scrum

The user is fully aware this isn't formal Agile delivery (no standups, no sprint reviews, no velocity tracking, no product owner ceremony). It's a solo developer using the mechanics that help and building them into a product feature at the same time.

**Future plan:** Detach the agile tooling from this project and build a dedicated agile platform as a separate product.

**How to apply:** Don't suggest Agile ceremony improvements or point out process gaps. The current approach is intentional and working. Treat the sprint/backlog system as a product feature being dog-fooded, not a process to optimise.
