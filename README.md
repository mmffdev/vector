# Vector PM

Enterprise agile / portfolio management platform. Private repo, Mac-only dev.

## Bring-up on a fresh Mac

**Prereqs:** Node 20+, Go 1.22+ (only if rebuilding the backend binary), Docker Desktop (only if running E2E tests), SSH key `~/.ssh/id_ed25519` with access to `mmffdev.com`.

```bash
git clone git@github.com:mmffdev/vector.git "MMFFDev - PM"
cd "MMFFDev - PM"
npm install
```

Env files (`backend/.env.local`) are committed — no manual setup.

### Quickest path: the launcher

Double-click `MMFF Vector Dev.app` — it brings up all three local services in one go: SSH tunnel (`localhost:5434` → remote Postgres), Go backend (`:5100`), Next.js frontend (`:5101`). See [`docs/c_dev-launcher.md`](docs/c_dev-launcher.md).

### Manual path

```bash
# 1. SSH tunnel to Postgres (one-off setup adds an `mmffdev-pg` alias)
./dev/scripts/ssh_manager.sh        # appends SSH config + opens tunnel
# subsequent sessions:
ssh -N -f mmffdev-pg                # localhost:5434 → remote :5432

# 2. Frontend (new terminal, repo root)
npm run dev                          # http://localhost:5101

# 3. Backend (new terminal)
cd backend
./server                             # tracked arm64 binary — http://localhost:5100
# or rebuild from source:
go build -o server ./cmd/server && ./server
```

If macOS blocks the tracked binary on first run: `xattr -dr com.apple.quarantine backend/server`.

### Optional: E2E tests

Selenium runs in a Docker container that drives a real browser against the live dev server.

```bash
docker run -d --name Selenium-Vector \
  -p 4444:4444 -p 7900:7900 \
  --shm-size 2g \
  selenium/standalone-all-browsers:nightly

npm run e2e                          # node:test runner; specs in e2e/
```

Watch the browser live at `http://localhost:7900` (password `secret`); Grid UI at `http://localhost:4444/ui/`. See [`docs/c_selenium.md`](docs/c_selenium.md) and [`dev/planning/plan_selenium_e2e.md`](dev/planning/plan_selenium_e2e.md).

## One-command laptop bootstrap

After `git clone` on any Mac (including a second/third machine), a single command gets you working:

```bash
./dev/scripts/bootstrap_laptop.sh
```

It runs three things in order, all idempotent — safe to re-run anytime:

1. **`link_memory.sh`** — creates the symlink so Claude Code's auto-memory reads from this repo.
2. **`ssh_manager.sh`** — installs brew / libpq / autossh / node@20, generates the SSH key, pushes it to `mmffdev.com`, appends `mmffdev-*` aliases to `~/.ssh/config`, starts the tunnel, writes `backend/.env.local`, runs `npm install`, verifies the DB round-trip.
3. **`install_launchagent.sh`** (asks first) — registers a LaunchAgent so the tunnel auto-starts at login and restarts on failure. Logs at `~/Library/Logs/mmff/tunnel.*.log`.

Full reference: [`docs/c_laptop_bootstrap.md`](docs/c_laptop_bootstrap.md).

### What's portable across laptops (via git)

| Thing | In repo | Per-laptop setup |
|---|---|---|
| `backend/.env.local` (DB password etc.) | ✅ | pulled |
| Auto-memory (`.claude/memory/`) | ✅ | symlink via `link_memory.sh` |
| Docs + `.claude/CLAUDE.md` | ✅ | pulled |
| SSH key `~/.ssh/id_ed25519` | ❌ | generated by `ssh_manager.sh` |
| `~/.ssh/config` aliases | ❌ | appended by `ssh_manager.sh` |
| brew packages, Node 20 | ❌ | installed by `ssh_manager.sh` |
| LaunchAgent plist | template in repo | installed by `install_launchagent.sh` |
| Tunnel process | ephemeral | auto-starts via LaunchAgent |

### Auto-memory round-trip

Claude Code's memory dir (`~/.claude/projects/.../memory/`) is a symlink to `.claude/memory/` in the repo. Saving a memory on laptop A writes a file in the repo; commit + push; pull on laptop B and the memory is immediately visible to Claude there. No per-machine re-learning.

### Database via web UI (Adminer)

With the tunnel running, open [http://localhost:8081](http://localhost:8081):

- **System**: PostgreSQL
- **Server**: `mmff-ops-postgres` (pre-filled)
- **Username**: `mmff_dev`
- **Password**: from `backend/.env.local` → `DB_PASSWORD`
- **Database**: `mmff_vector`

Container lifecycle and security posture: [`docs/c_c_adminer.md`](docs/c_c_adminer.md).

### LaunchAgent — auto-tunnel at login

```bash
launchctl bootout  gui/$(id -u)/com.mmff.tunnel                        # stop
launchctl bootstrap gui/$(id -u) ~/Library/LaunchAgents/com.mmff.tunnel.plist  # start
tail -f ~/Library/Logs/mmff/tunnel.err.log                             # watch
```

The agent runs `ssh -N mmffdev-pg` with `KeepAlive` scoped to network state and a 30-second throttle, so it waits for network before relaunching and doesn't busy-loop on repeated failures.
