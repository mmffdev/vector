Database Architecture

Two separate Postgres databases serve this system. They are never joined — cross-DB references are app-enforced (UUIDs stored but no DB-level FK across servers).

mmff_library (read-only at runtime)

MMFF-authored content. Padmin cannot edit it. The app reads here to list and fetch models.

  • portfolio_models — spine record; versioned
  • portfolio_model_layers — hierarchical layer tree
  • portfolio_model_workflows — per-layer state machine states
  • portfolio_model_workflow_transitions — allowed transitions between states
  • portfolio_model_artifacts — feature flags (board, sprint, PI planner)
  • portfolio_model_terminology — label overrides (e.g. "Epic" → custom word)
⇄
mmff_vector (tenant data, read-write)

Per-subscription operational data. The adoption saga writes here; live product reads here.

  • subscription_portfolio_model_state — adoption lifecycle record
  • subscription_layers — mirror of adopted layers
  • subscription_workflows — mirror of adopted workflow states
  • subscription_workflow_transitions — mirror of transition edges
  • subscription_artifacts — mirror of enabled features/config
  • subscription_terminology — mirror of label overrides
Why mirrors? The library DB is managed by MMFF (read-only to tenants). When a padmin adopts a model, every relevant row is copied into the vector DB under their subscription_id. After adoption, zero queries hit the library DB for day-to-day operation. Mirrors also capture the model version at adoption time via source_library_version, so library updates don't silently change live tenant configs.

Step 1 — Padmin Logs In & Sees Available Models

padmin only   The portfolio model selection UI is gated to the padmin role. A regular user or gadmin cannot access this flow.

1

Auth & Session

  • Padmin posts credentials to POST /api/auth/login
  • Backend checks mmff_vector.users table: WHERE email=$1 AND subscription_id=$2
  • JWT issued — claims include subscription_id, user_id, role="padmin"
  • All subsequent requests carry Authorization: Bearer <jwt>
2

Check Adoption State

  • Frontend calls GET /api/portfolio-models/adoption-state
  • Reads mmff_vector.subscription_portfolio_model_state
  • Query: WHERE subscription_id=$1 AND status='completed' AND archived_at IS NULL
  • If adopted → redirected to adopted model view
  • If not adopted → wizard shown: model card list
3

Fetch Available Models

  • Frontend calls GET /api/portfolio-models
  • Backend reads mmff_library portfolio_models + portfolio_model_layers
  • Query filters: scope='system', visibility='public', archived_at IS NULL
  • Returns: name, description, layer count, layer name summary, version, model_family_id
  • No write to any DB at this point
4

Model Detail (on hover/click)

  • Frontend calls GET /api/portfolio-models/{id}
  • Backend opens REPEATABLE READ + READ ONLY transaction on mmff_library
  • Reads all 6 tables for that model (spine + 5 children) in one consistent snapshot
  • Returns Bundle struct — layers, workflows, transitions, artifacts, terminology
  • No write to any DB
API Call DB Table(s) Read DB Write?
POST /api/auth/login vector users No (token issued in memory)
GET /api/portfolio-models/adoption-state vector subscription_portfolio_model_state No

Step 2 — Padmin Selects the Model and Confirms

Padmin clicks a model card and hits Confirm/Adopt. This triggers the HTTP POST that starts the saga. Before any DB write, the backend runs idempotency checks.

1

POST /api/portfolio-models/{id}/adopt

  • Backend extracts model_id from URL path
  • Extracts subscription_id + user_id from JWT claims
  • Calls Orchestrator.Adopt()
2

Idempotency Check

  • Reads subscription_portfolio_model_state for this subscription
  • status=completed, same model → return 200 immediately (no saga, idempotent)
  • status=completed, different model → return 409 conflict
  • status=in_progress → return 409 conflict (already running)
  • status=failed, same model → resume saga from failed step
  • status=failed, different model → archive stale row, start fresh
  • no row → start fresh saga
3

Library Snapshot (Read-Only)

  • Opens REPEATABLE READ + READ ONLY tx on mmff_library
  • Reads all 6 tables for the selected model into memory as a Bundle
  • This snapshot is used for ALL subsequent saga steps — library is never re-queried mid-saga
  • Tx closed (read-only, no commit needed)
4

First Vector DB Write

  • Opens write tx on mmff_vector
  • INSERT into subscription_portfolio_model_state: status='in_progress'
  • Row created — this is the adoption record that tracks the saga lifecycle
  • Returns to SSE stream (step events begin flowing)

Already completed, same modelcompletedNo write200 OK immediately

Step 3 — Adoption Saga (7 Steps via SSE Stream)

The saga runs synchronously server-side but results are streamed to the browser via Server-Sent Events (SSE). Each step fires a step event; the overlay animates the card for that step. All writes go to mmff_vector; all reads come from the in-memory Bundle snapshot taken in step 2.

SSE Transport: The frontend uses fetch() + ReadableStream (not EventSource) so it can send Authorization: Bearer <jwt> to the backend at http://localhost:5100 (port 5100, not through Next.js). The SSE endpoint is GET /api/portfolio-models/{id}/adopt/stream.
1

step: validate

  • library Confirms model exists and is fetchable
  • Uses the in-memory Bundle snapshot (no new DB query)
  • vector INSERT into subscription_portfolio_model_state
  • Writes: subscription_id, adopted_model_id, adopted_by_user_id, status='in_progress'
  • SSE: event: step → {"index":0,"name":"validate","status":"ok"}
2

step: layers

  • Reads layers from Bundle snapshot (library — no new query)
  • vector INSERT into subscription_layers
  • Two-phase: first insert root layers (no parent), then update parent_layer_id FKs for children using newly-assigned UUIDs
  • Copies: name, tag, sort_order, icon, colour, description_md, help_md, allows_children, is_leaf, source_library_id, source_library_version
  • ON CONFLICT DO NOTHING — idempotent
  • SSE: event: step → {"index":1,"name":"layers","status":"ok"}
3

step: workflows

  • Reads workflows from Bundle snapshot
  • Translates library layer UUID → mirror subscription_layer UUID (using map built in step 2)
  • vector INSERT into subscription_workflows
  • Copies: state_key, state_label, sort_order, is_initial, is_terminal, colour, layer_id (mirror FK)
  • Builds workflow UUID map for transitions step
  • SSE: event: step → {"index":2,"name":"workflows","status":"ok"}
4

step: transitions

  • Reads transition edges from Bundle snapshot
  • Translates library workflow UUIDs → mirror subscription_workflow UUIDs
  • vector INSERT into subscription_workflow_transitions
  • Copies: from_state_id (mirror), to_state_id (mirror), source_library_id, source_library_version
  • CHECK: from ≠ to enforced in migration
  • SSE: event: step → {"index":3,"name":"transitions","status":"ok"}
5

step: artifacts

  • Reads artifact toggles from Bundle snapshot
  • vector INSERT into subscription_artifacts
  • Copies: artifact_key, enabled (bool), config (JSONB)
  • Controls feature flags like board view, sprint planning, PI planner
  • SSE: event: step → {"index":4,"name":"artifacts","status":"ok"}
6

step: terminology

  • Reads label overrides from Bundle snapshot
  • vector INSERT into subscription_terminology
  • Copies: key (e.g. "epic_label"), value (e.g. "Initiative")
  • UI reads these to rename standard labels per model
  • SSE: event: step → {"index":5,"name":"terminology","status":"ok"}
7

step: finalize

  • vector UPDATE subscription_portfolio_model_state
  • Sets: status='completed', adopted_by_user_id=$user, adopted_at=NOW()
  • This is the commit point — the subscription is now officially on this model
  • SSE: event: step → {"index":6,"name":"finalize","status":"ok"}
  • SSE: event: done → {"state_id":"...","status":"completed","adopted_at":"..."}

Error Handling During Saga

Failure PointActionState Table Result
Any step (1–7) failSaga() called — UPDATE state row status='failed', INSERT into error_events with JSON context failed
SSE: fail event event: fail sent to browser → {"step":"...","error_code":"ADOPT_*","message":"..."} UI shows error state
Retry same model Finds failed row + same model → resetFailedToInProgress(), saga resumes in_progress → completed

Idempotent Completed Path (Already Adopted)

Idempotency: If the padmin hits adopt on an already-completed model, Adopt() returns immediately with the existing result — no saga runs, zero step events fire. The SSE handler detects stepCount == 0 on a successful result and emits 7 synthetic ok step events before the done terminator, so the overlay still animates correctly.

Step 4 — Live State After Adoption Is Complete

Once status='completed' is set, the subscription runs entirely off its mirror tables in mmff_vector. The library DB is not consulted again unless a re-adoption is triggered.

subscription_portfolio_model_state
One active row per subscription. status='completed', archived_at IS NULL. Unique partial index ensures only one live adoption.
✓ adopted_model_id = library UUID
subscription_layers
N rows — one per layer in the adopted model. parent_layer_id is a self-FK within this table (not the library). Hierarchical tree fully local.
✓ source_library_id, source_library_version captured
subscription_workflows
M rows — one per state per layer. layer_id FK → subscription_layers. is_initial / is_terminal flags drive UI workflow rendering.
✓ state_key, state_label, colour copied
subscription_workflow_transitions
T rows — directed edges between workflow states. Both from_state_id and to_state_id are FKs into subscription_workflows (CASCADE on delete).
✓ from/to FKs resolve locally
subscription_artifacts
Feature toggle rows. artifact_key (e.g. "board", "sprint") + enabled bool + config JSONB for per-feature settings.
✓ enabled/config govern feature availability
subscription_terminology
Label override rows. UI reads these to substitute model-specific language ("Epic" → "Initiative", etc). Keyed by string key.
✓ key/value pairs for UI label substitution

Step 5 — Dev Reset Flow

gadmin or padmin   The Dev Setup page exposes a Reset Adoption State button. It is internal tooling — it wipes all mirror data for the caller's subscription back to zero, as if adoption never happened.

Scope: The reset is always scoped to the caller's own subscription_id (from the JWT). A gadmin cannot reset another subscription's data through this endpoint. The filter source_library_id IS NOT NULL on mirror tables protects any hand-authored rows (if any exist in future) from accidental deletion.

Endpoint

POST /api/admin/dev/adoption-reset — requires gadmin or padmin role

Deletion Sequence (children deleted before parents)

  • subscription_artifacts
    DELETE FROM subscription_artifacts WHERE subscription_id=$1 AND source_library_id IS NOT NULL
  • subscription_terminology
    DELETE FROM subscription_terminology WHERE subscription_id=$1 AND source_library_id IS NOT NULL
  • subscription_workflow_transitions
    DELETE FROM subscription_workflow_transitions WHERE subscription_id=$1 AND source_library_id IS NOT NULL
    Must go before workflows — from_state_id/to_state_id FK into workflows
  • subscription_workflows
    DELETE FROM subscription_workflows WHERE subscription_id=$1 AND source_library_id IS NOT NULL
    Must go before layers — layer_id FK into subscription_layers
  • subscription_layers
    DELETE FROM subscription_layers WHERE subscription_id=$1 AND source_library_id IS NOT NULL
    Parent rows (self-referential parent_layer_id) — children already gone by cascade via workflows delete
  • subscription_portfolio_model_state — NO filter on source_library_id
    DELETE FROM subscription_portfolio_model_state WHERE subscription_id=$1
    Wipes all rows including failed/rolled_back history for this subscription

After Reset: What Padmin Sees

On next page load, GET /api/portfolio-models/adoption-state returns {"adopted":false}. The wizard re-appears. Padmin can adopt any model — including the same one again — as if it's a first adoption. The full saga runs again and re-populates all mirror tables from scratch.

Adoption Status Enum

Column: subscription_portfolio_model_state.status TEXT CHECK constraint enforces exactly these values.

StatusWhen SetUnique Index Active?Can Padmin Adopt Again?
pending Reserved for future two-phase commit pattern. Not currently used — saga skips directly to in_progress. Yes (blocks new adoption) No — 409 conflict
in_progress Set immediately when saga starts (validate step). Clears to completed or failed. Yes (blocks new adoption) No — 409 conflict
completed Set in finalize step. This is the live/healthy state. Yes (blocks new adoption) Only if same model (idempotent 200)
failed Set by failSaga() on any step error. Mirror tables may be partially written. No — excluded from unique index Yes — same model resumes; different model archives stale row and starts fresh
rolled_back Reserved for future re-adoption / model switch. Not yet implemented. No — excluded from unique index Yes (treated same as failed)
Unique index rule: idx_subscription_portfolio_model_state_active_unique is a partial unique index on (subscription_id) WHERE archived_at IS NULL AND status NOT IN ('failed', 'rolled_back'). This allows one failed row + one in_progress row to coexist briefly during the archive-and-restart sequence, but guarantees only one "live" adoption per subscription at any time.
