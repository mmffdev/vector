# PLA-0042 — Chrome scope picker (active topology scope)

**Feature area:** FE-POR-0002 (scope / role-grant chrome surface)
**Status:** Implemented — picker UI + ScopeContext only; backlog/portfolio read wiring deferred.
**Research:** [R051 — Project / Scope Picker UX](dev/research/R051.json) (Rally / Jira / ADO / Linear).
**Related:** PLA-0006 (topology), PLA-0007 (roles / permissions).

---

## Why this plan exists

Topology nodes are how a tenant carves their organisation into addressable scopes (offices, programs, teams). Admins grant users an `admin | editor | viewer` role on a node via `topology_role_grants` — and that grant set is the source of truth for *what work this user can see*. Until this plan, no chrome surface let the user **pick** between the scopes they've been granted; the only place a "level" was visible was the admin-side topology editor, which is the wrong audience.

R051 surveyed Rally / Jira / ADO / Linear and converged on the top-left chrome dropdown with URL-persisted scope. This plan implements that pattern.

---

## Scope of THIS plan (Iteration 1)

**In:**
- Backend endpoint `GET /api/topology/grants/me` returning every live grant on a live node held by the authed user, spanning workspaces in the subscription (not workspace-clamped — a user can hold grants in several workspaces and the picker must surface them all).
- Frontend `ScopeContext` holding `{grants, activeNodeId, activeGrant, loading, error, setActiveNodeId, reload}` with URL `?scope=<node_id>` + localStorage persistence; falls back to `null` if the persisted id no longer matches a live grant.
- Frontend `<ScopePicker />` chrome component mounted in `.page-header__left` next to the page title — trigger button + dropdown with filter input + indented tree of granted nodes + role pill per row.
- CSS in `app/globals.css` under `.scope-picker*` namespace; no inline styles; indent levels modelled as `__item--d0` … `__item--d6` modifier classes.

**Deferred to a follow-up plan (NOT in this PR):**
- Wiring `activeNodeId` into backlog / portfolio / work-item reads as a `?scope=` server filter. Today, picking a scope only persists the choice — it does not yet narrow what the user sees on `/work-items`, `/portfolio-items`, dashboards, etc.
- Grant-management UI for admins (already exists on the topology editor — `grantRole` / `revokeRole`).
- "Recent scopes" / "Pinned scopes" submenus (R051 nice-to-have; Linear / Jira show this).

## Edge cases handled

| Case | Behaviour |
|---|---|
| User has zero grants | Picker renders nothing (no disabled stub). Header stays clean. |
| `?scope=` points at a revoked / archived grant | Falls back to `null`. URL is rewritten on next `setActiveNodeId`. |
| localStorage disabled (private mode / quota) | Try/catch swallows; URL param is the canonical channel. |
| Grant tree spans >7 levels | Visual indent caps at depth 6 — labels stop nesting visually but stay readable. |
| Filter input matches nothing | "No matches." empty-state in the list area. |
| `listMyGrants` errors | Trigger label stays at "Select scope"; opening the panel shows the error inline. |

## Lint / writer-boundary

- No new tables. `topology_role_grants` writes still happen exclusively through `orgdesign.Service` (PLA-0006 sole-writer); the new `ListMyGrants` is a **read**, so `lint:writer-boundary` is unaffected.
- No new addressable substrate slots — the picker lives in the page header which is already an addressable region.
- No raw `<table>`; no inline `style={{}}`; no `dui-*` classes (this is user chrome, not Dev).

## Verification

1. `npx tsc --noEmit` — no new errors against new files.
2. `go build ./internal/orgdesign/... ./cmd/server/...` — compiles.
3. Manual: log in as a user with ≥1 grant. Picker appears under the page title. Open it; the granted nodes appear as an indented tree. Pick one — URL becomes `?scope=<id>`, picker label updates, localStorage key `vector.scope.activeNodeId` is set.
4. Hard refresh — picker rehydrates from URL.
5. Strip the `?scope=` query manually but keep localStorage — picker rehydrates from localStorage.
6. Revoke the active grant via admin → reload — picker falls back to "Select scope".
7. Log in as a user with zero grants — picker is absent (not a disabled stub).

## Follow-up — Iteration 2 (separate plan)

Wire `activeNodeId` into the backlog / portfolio / dashboard read paths as a `?scope=<id>` filter. That work needs its own scope decomposition because every consuming endpoint (work-items, portfolio-items, sprints, releases) gets a clamp parameter and the backend has to decide whether scope clamps stack with workspace clamps or replace them. Out of band for the picker itself.
