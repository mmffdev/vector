# PLA-0041 — Orbit View transition editor + artefact-move enforcement

**Feature area:** FE-GOV-0004
**Page moves:** `/workspace-settings/customisation/flow-states` → `/workspace-settings/workspace-settings/flow-states` and `/workspace-settings/customisation/work-items` → `/workspace-settings/workspace-settings/work-items` (both leave the Customisation L3 group and join the Workspace Settings L3 group as siblings of Organisation / Workspaces / Custom Fields / Portfolio Model).
**New page:** `/workspace-settings/workspace-settings/transition-rules` — slots between Flow States and Work Items, hosts the Orbit View editor per flow.
**Replaces:** the N×N `TransitionMatrix` previously on the Flow States page — Flow States becomes pure state/colour/description/exit-rules editor; transition rules move to their own page so the cognitive load is split.
**Related:** PLA-0040 (descriptions + exit rules — same flow trio), FLOW1, F1.
**Status:** Draft — awaiting user approval before any code is written.

---

## Why this plan exists

The user dropped a complete design brief (`Flow State Journey Maker.md`) for an "Orbit View" component to replace the existing N×N transition matrix, and asked that we also *"ensure artefacts can move across our state correctly"*. That second clause is a real backend gap, not just a UI swap.

**Two distinct workstreams, one plan because they share the same governance surface:**

1. **UI** — replace the matrix with the Orbit View (`Where can a card go from HERE?` — one source state at a time, destinations orbiting around it).
2. **Enforcement** — the backend currently checks that a target `flow_state_id` *exists* but does **not** check that the `(current_state → new_state)` edge is in `flow_transitions`. So a user can define rules in the matrix/orbit view and the system silently ignores them. The rules are advisory, not enforced. This plan closes that hole.

The two streams ship together because shipping just the editor without enforcement makes the editor a lie; shipping just enforcement without a usable editor is a support nightmare.

## Workstream A — New Transition Rules page (presentation surface)

### Scope

Stand up a **new page** at `/workspace-settings/customisation/transition-rules` that hosts the Orbit View editor for every flow in the workspace. **Delete** the existing `TransitionMatrix` from [`flow-states/page.tsx:1159-1246`](app/(user)/workspace-settings/customisation/flow-states/page.tsx#L1159-L1246) — the Flow States page no longer owns transition editing. **No backend changes** — `flowStatesApi.createTransition` / `deleteTransition` already exist and are reused unchanged.

**Why a separate page** (user decision): the Flow States page already carries state colour, description, exit rules, kind, is_pullable, and ordering. Adding the Orbit View on top would overload the page and conflate two distinct mental models — *"what does this state mean?"* (flow states) vs *"how can a card move between states?"* (transition rules). One page per question.

## Workstream B — Artefact-move enforcement (the real fix)

### The gap

[`backend/internal/artefactitemsv2/service.go:675-693`](backend/internal/artefactitemsv2/service.go#L675-L693) validates only that the target `flow_state_id` exists and belongs to the right subscription:

```go
if in.FlowStateID != nil {
    // Validate the flow_state belongs to this subscription.
    var fsExists bool
    err := s.vectorArtefactsPool.QueryRow(ctx, `
        SELECT EXISTS(
            SELECT 1 FROM flow_states fs
            JOIN flows f ON f.id = fs.flow_id
            JOIN artefact_types at ON at.id = f.artefact_type_id
            WHERE fs.id = $1 AND at.subscription_id = $2 AND fs.archived_at IS NULL
        )`, *in.FlowStateID, subscriptionID,
    ).Scan(&fsExists)
    if err != nil || !fsExists {
        return nil, fmt.Errorf("%w: flow_state_id not found", ErrInvalidInput)
    }
    // … then writes the new state with no rule check
}
```

So if a workspace defines `Backlog → To Do` and `To Do → Doing` as the only allowed transitions, the API will still happily accept `Backlog → Done` — making the editor cosmetic.

### Fix

In the same `if in.FlowStateID != nil` block, before the `sets = append…` line, look up the *current* `flow_state_id` for the artefact, then check `flow_transitions` for the `(current, new)` edge. Two SQL options:

**Option 1 — separate read then check (clearer, two round-trips):**
```go
var currentStateID uuid.UUID
err = s.vectorArtefactsPool.QueryRow(ctx,
    `SELECT flow_state_id FROM artefacts
     WHERE id = $1 AND subscription_id = $2 AND archived_at IS NULL`,
    id, subscriptionID).Scan(&currentStateID)
if err != nil { return nil, ErrNotFound }

if currentStateID != *in.FlowStateID {
    var allowed bool
    err = s.vectorArtefactsPool.QueryRow(ctx,
        `SELECT EXISTS(
           SELECT 1 FROM flow_transitions
           WHERE from_state_id = $1 AND to_state_id = $2
         )`, currentStateID, *in.FlowStateID).Scan(&allowed)
    if err != nil { return nil, err }
    if !allowed {
        return nil, fmt.Errorf("%w: transition not permitted", ErrInvalidInput)
    }
}
```

**Option 2 — single query, less reasoning needed:**
```sql
SELECT EXISTS(
  SELECT 1
  FROM artefacts a
  WHERE a.id = $1 AND a.subscription_id = $2 AND a.archived_at IS NULL
    AND (
      a.flow_state_id = $3  -- no-op move; allow
      OR EXISTS (
        SELECT 1 FROM flow_transitions ft
        WHERE ft.from_state_id = a.flow_state_id AND ft.to_state_id = $3
      )
    )
)
```

**Recommendation:** Option 1. Easier to log, easier to return distinct error messages for `(not found)` vs `(transition not permitted)`, and the second query only fires when the state actually changes.

### Edge cases the enforcement must handle

| Case | Behaviour |
|---|---|
| `current == new` (no-op move) | Allow — never check `flow_transitions`. Saves us from having to define self-edges. |
| Initial-create (no `current_state` yet) | Out of scope here; `PatchWorkItem` only runs after create. Create uses the flow's `is_initial` state, no transition check needed. |
| `flow_transitions` table empty for this flow | This is the *brand-new workspace* state — every state is reachable from any state by default? Or every state is reachable from none? **Decision needed**: default to "if no transitions defined at all for the flow, allow any move". This matches the existing UX where a fresh workspace doesn't need to set up rules before moving cards. Implemented as: skip the `EXISTS` check when `(SELECT COUNT(*) FROM flow_transitions ft JOIN flow_states fs ON fs.id = ft.from_state_id WHERE fs.flow_id = (… flow of current state …)) = 0`. |
| Transition rule deleted while a card was mid-state | Card stays where it is. No retroactive enforcement. Future moves are gated by the new rule set. |
| Cancellation / archive moves | Out of scope for transition rules — those go through dedicated archive endpoints, not `PatchWorkItem.FlowStateID`. |

The empty-flow default needs sign-off — flag for user in the post-plan conversation, default behaviour above is the safest.

### Error surface

New error message returned via the same `ErrInvalidInput` family — frontend already handles these via `notify.apiError`. The toast becomes *"Move not allowed: transition not permitted"* (or similar — copy in the handler, not the service).

### Same gap exists in… (audit before plan execution)

The fix has to land everywhere `flow_state_id` can be written. Confirm with grep before implementation, but at minimum:

- `backend/internal/artefactitemsv2/service.go` (`PatchWorkItem`) — confirmed gap, fix here.
- `backend/internal/portfolioitemsv2/service.go` — strategy/portfolio items. Likely has the same gap. Audit + fix if so.
- Any bulk-move endpoint (`/bulk-status` style). Audit.
- Any drag-to-column kanban endpoint. Audit.

**Don't fix one and miss the others** — a single uncovered writer turns the enforcement off for that path.

1. **Empty-flow default**: *"if no transitions are defined for this flow, allow any move"* — recommended for back-compat. Alternative: *"if no transitions defined, allow no moves except self"* — strict but breaks fresh-workspace UX. **Recommend: open default**.
2. **Portfolio items** — confirm we want the same enforcement on strategy items, or treat them as advisory-only for now. **Recommend: same enforcement** (consistency > scope creep).
3. **Replace vs co-exist** — drop `TransitionMatrix` entirely on this PR, or keep it under a `?view=matrix` query param for one release? **Recommend: drop entirely** (no users have it in muscle memory; matrix is internal-only and the brief explicitly says *"This replaces the existing 'Allowed transitions' matrix screen"*).
4. **Working-prototype reuse**: the brief references `Flow rule builders.html` → artboard `03 · Orbit view` + `m3-orbit.jsx`. Confirm those files exist and we should copy the SVG math verbatim, or treat the brief's math snippet as the spec.
